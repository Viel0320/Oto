总评

架构方向正确

项目已经从旧式大仓库向 LibraryFacade、细粒度 Gateway、Graph 组合根、Typed dependency views 迁移。 ContainerAccessArchitectureTest、GraphLifecycleTest、PlaybackDependencyViewArchitectureTest 说明团队已经在用测试守住 interface，而不是只依赖约定。

生产适配仍不稳

真实发布前最危险的不是功能代码，而是配置与持久化：android:permission="TODO"、 fallbackToDestructiveMigration(true)、DataStore 凭据备份遗漏、入库 release keystore 与明文密码。 这些问题会绕过大部分单元测试。

深模块还没完全形成

多数 module 已经被切开，但部分 interface 仍然 shallow：LibraryFacade 是全量委托总线， APlayerApp 是 UI shell 协调中心，播放状态跨 Manager/Service/Handler/DataSource 分散。

Problem: schema 已到 40，仓库保留了 28-40 的 schema 文件，但实现只声明了 36->37 和 39->40，其他路径依赖 destructive fallback。对有声书播放器来说，进度、书签、ABS mirror、根目录授权状态都是长期资产，不能默认丢失。

Benefits: interface 变小：调用方只知道“数据库可升级”；实现内部吸收历史版本细节。locality 集中在迁移 module；leverage 是每次 schema 变更都能复用同一套回归测试。

  • app/src/main/java/com/viel/aplayer/media/VfsPlaybackDataSource.kt:18,56,63,105,112,139

Problem: runBlocking 内部 watcher 在 Dispatchers.Default 协程里调用 Thread.currentThread()，很可能拿到的是 watcher 自己的线程，而不是 Media3 loader 线程。这样取消/切歌/seek 中断时，旧的 DB lookup 或 WebDAV open 可能不会及时停下。另一个风险是 open 阶段 IOException 被映射为 ERROR_CODE_IO_READ_POSITION_OUT_OF_RANGE，网络失败和真正越界会混在一起。

Solution: 提取 BlockingVfsOpenAdapter 或等价内部 module，在进入 runBlocking 前捕获 loader thread；统一映射 InterruptedIOException、timeout、not found、range unsupported。补一个可控 fake reader：阻塞 open 后中断 loader thread，断言立即取消并返回标准异常。

Benefits: locality 集中在播放 I/O seam；leverage 是 ExoPlayer、WebDAV、ABS streaming、SAF open 共享同一取消语义；测试面从真实网络变成 fake adapter。

Problem: LibraryFacade 删除测试看起来仍会把复杂度分散到多个 ViewModel，因此它是有价值的；但它通过 Kotlin delegation 暴露所有 gateway，interface 接近所有实现能力之和。长期看，新功能会继续往 facade 上挂方法，形成“表面不大、实际全知”的 shallow module。

Top recommendation

先做 Phase 0：生产策略止血

当前项目的架构重构方向基本正确，不建议马上继续扩大结构性拆分。第一优先级应是配置层和发布层： PlaybackService 权限、DataStore 凭据备份、release signing、cleartext/TLS 规则、乱码文案测试。 这些改动 scope 小、可回归、收益高，并且能防止后续架构优化建立在不可靠的生产基础上。
